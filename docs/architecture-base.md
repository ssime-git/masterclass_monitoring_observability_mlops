# Base Application Architecture

## From Requirements to Architecture

On `main`, we defined the business need and the full list of requirements. This document explains how the architecture in branch `01-architecture-base` implements them.

The application classifies short support-style messages into three categories: `billing`, `technical`, `account`. The goal is not to train a model. The goal is to understand how a small ML-enabled application is structured, protected, and prepared for operational visibility.

## Requirements Addressed in Branch 01

The functional and non-functional requirements are defined in the [masterclass outline](masterclass-outline.md). Here is how the architecture addresses them:

**Functional requirements** (all implemented in branch 01):

- Authentication and sessions: handled by the gateway, persisted in SQLite
- Document classification: handled by the model service, called through the gateway
- Prediction history: stored per session in SQLite, returned with each classification
- Access control: the gateway rejects unauthenticated requests before they reach the model

**Non-functional requirements** (partially implemented in branch 01):

- Rate limiting: NGINX applies it at the edge before requests reach the gateway
- Service isolation: each service has a single responsibility and clear boundaries
- Metrics endpoints: already exposed by the gateway and model service, activated by Prometheus in branch 02
- Structured logs and traces: activated in this branch (03) with OpenTelemetry, Loki, and Tempo
- Local inspectability: SQLite is mounted from `data/`, readable from the host

## Services and Responsibilities

- `streamlit-ui`: user-facing interface for login and document classification
- `nginx`: reverse proxy and rate limiting layer
- `gateway`: authentication, session validation, orchestration, and persistence access
- `model-service`: inference endpoint for the classifier
- `SQLite`: local database for users, sessions, and prediction history

## Request Lifecycle

1. The user opens Streamlit and submits credentials.
2. Streamlit sends the login request through `nginx`.
3. `nginx` forwards the request to the gateway and enforces rate limiting.
4. The gateway validates the credentials against SQLite.
5. The gateway creates a session token and returns it to the client.
6. Streamlit stores that token and reuses it for authenticated API calls.
7. For classification, the gateway validates the session, forwards the text to the model service, stores the result history, and returns the response.

## Security and State Boundaries

- Session tokens are generated by the gateway, not by the UI.
- The UI only stores and reuses the returned bearer token.
- Rate limiting happens before the gateway, at the `nginx` layer.
- SQLite is mounted from `data/`, so the application state remains inspectable from the host machine.

## Local Commands

Install and validate:

```bash
make install
make lint
make typecheck
make test
```

Start and stop the stack:

```bash
make up
make down
```

Open after startup:

- Streamlit UI: `http://localhost:8501`
- Public API through NGINX: `http://localhost:8080`

## Reproduce the Main Flow

Create a session:

```bash
curl -s http://localhost:8080/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"alice","password":"mlops-demo"}'
```

Call the protected classify endpoint with the returned token:

```bash
TOKEN="$(curl -s http://localhost:8080/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"alice","password":"mlops-demo"}' \
  | python3 -c 'import sys, json; print(json.load(sys.stdin)["access_token"])')"

curl -s http://localhost:8080/api/classify \
  -H "Authorization: Bearer ${TOKEN}" \
  -H 'Content-Type: application/json' \
  -d '{"text":"My payment failed and I need a refund."}'
```

Inspect persisted state:

```bash
ls -lh data/
sqlite3 data/masterclass.db '.tables'
```
